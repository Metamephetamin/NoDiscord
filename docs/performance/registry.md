# Performance Registry

Единый реестр лагов и точек оптимизации для первой волны `Electron Windows`.

Шаблон записи:
- `id`
- `зона`
- `сценарий`
- `симптом`
- `шаги воспроизведения`
- `baseline`
- `целевая метрика`
- `гипотеза`
- `приоритет`
- `статус`
- `связанный фикс`

Severity по умолчанию:
- `P0`: повторяемый фриз `> 500 ms` или блокирующая задержка.
- `P1`: заметная задержка `200–500 ms`.
- `P2`: микролаг `100–200 ms` или явная деградация плавности.
- `P3`: всё, что ниже, но системно создаёт ощущение неотзывчивости.

## PERF-001 - Медленное открытие batch upload окна в TextChat
- Зона: `text-chat`
- Сценарий: выбор `3+` изображений из проводника перед отправкой.
- Симптом: окно подготовки группы фото открывается с заметной задержкой, интерфейс выглядит подвисшим.
- Шаги воспроизведения: открыть чат, выбрать пачку изображений через file picker, дождаться появления окна подготовки.
- Baseline: по пользовательскому наблюдению `~2-5 s` до первого внятного состояния окна.
- Целевая метрика: `<= 150 ms` до первого видимого состояния и без блокировок main thread `> 100 ms`.
- Гипотеза: лишние ререндеры chat tree, тяжёлое создание превью в критическом пути, layout thrash при открытии оверлея.
- Приоритет: P0
- Статус: open
- Связанный фикс: `perf-tracing + deferred preview hydration`

## PERF-002 - Oversized renderer bundle
- Зона: `app-shell`
- Сценарий: cold start и первый вход в основное окно.
- Симптом: тяжёлый первичный рендер и риск поздней интерактивности из-за крупного основного renderer bundle.
- Шаги воспроизведения: выполнить `npm run build:frontend`, проверить размер главного renderer chunk и холодный старт packaged Electron.
- Baseline: основной renderer JS chunk `~1.27 MB` minified по текущей сборке.
- Целевая метрика: уменьшить общий eager JS footprint и разложить редко используемые зоны на lazy chunks.
- Гипотеза: слишком много workspace/chat/media/voice логики попадает в основной bundle раньше времени.
- Приоритет: P1
- Статус: open
- Связанный фикс: `bundle split / lazy init`

## PERF-003 - Очень тяжёлый noise suppression chunk
- Зона: `voice`
- Сценарий: старт приложения и инициализация voice stack.
- Симптом: отдельный очень крупный JS chunk под voice/noise suppression создаёт риск лишней ранней загрузки и тяжёлой инициализации.
- Шаги воспроизведения: выполнить `npm run build:frontend`, проверить размер `noise_suppression` chunk и его eager/lazy загрузку.
- Baseline: chunk `noise_suppression` `~4.82 MB` minified по текущей сборке.
- Целевая метрика: отложенная загрузка voice-heavy зависимостей до реального входа в звонок/голосовую комнату.
- Гипотеза: voice stack инициализируется раньше реальной необходимости.
- Приоритет: P1
- Статус: open
- Связанный фикс: `lazy voice init`

## PERF-004 - MenuMainController перегружен состояниями и вычислениями
- Зона: `menu-main`
- Сценарий: переключение workspace, серверов, каналов, открытие настроек и оверлеев.
- Симптом: большой централизованный controller повышает шанс массовых ререндеров и тяжёлых переходов.
- Шаги воспроизведения: открыть основное меню, часто переключать секции, серверы и настройки, смотреть perf traces и long tasks.
- Baseline: `MenuMainController.jsx` около `212 KB`, один из крупнейших файлов проекта.
- Целевая метрика: сократить рендер churn и вынести независимые state islands из общего горячего дерева.
- Гипотеза: слишком много UI-состояния живёт в одном контроллере и затрагивает лишние подсистемы.
- Приоритет: P1
- Статус: open
- Связанный фикс: `state isolation / memo boundaries`

## PERF-005 - TextChat tree чувствителен к локальным state update
- Зона: `text-chat`
- Сценарий: отправка вложений, открытие media preview, reply/edit, длинный скролл.
- Симптом: локальные изменения composer/media state потенциально тянут за собой лишние обновления большого chat tree.
- Шаги воспроизведения: открыть длинный чат, переключать reply/edit/media preview, выбирать вложения, смотреть perf traces и long tasks.
- Baseline: есть уже подтверждённые проблемы с повторным рендером и открытием media/upload flow.
- Целевая метрика: локальные UI-операции не должны вызывать массовый пересчёт ленты сообщений.
- Гипотеза: недостаточно изолированы message list, composer, media overlays и upload queue.
- Приоритет: P0
- Статус: in_progress
- Связанный фикс: `memoized message list + perf traces`
